# 多集群网格迁移

本章介绍多集群场景下，将 Mesh v1 并存灰度迁移到 Mesh v2 时相对单集群流程的叠加内容。多集群迁移以[单集群网格迁移](../migrating-a-single-cluster-mesh/index.mdx)中描述的 P0 至 P6 六阶段流程为基线，本章各篇只描述多集群场景下需要叠加的步骤与差异点，不重复单集群流程中已经说明的内容——开始阅读本章之前，请确认已经完整阅读[单集群网格迁移](../migrating-a-single-cluster-mesh/index.mdx)。

多集群网格的迁移路径按现有 Mesh v1 环境的网络拓扑分为两条：跨集群 Pod 网络能够直接互通的**单网络（single network）**拓扑，与依赖东西向网关转发跨集群流量的**多网络（multi network）**拓扑。两条路径在迁移期间与迁移终态对 network 相关配置的处理方式截然不同，进入具体的操作页面之前，必须先完成下面的判定。

## 判定网络拓扑

判定过程分两步：先采集两个业务集群各自的网格标识，再执行一次跨集群连通性检查；将两部分结果代入下文的判定表，即可确定当前环境应当遵循哪一条迁移路径。判定只需要执行一次，其采集结果（尤其是 meshId 与两侧的 network 取值）在后续迁移步骤中还会被复用，建议记录留存。

<Steps>

### 采集两集群的网格标识

对两个业务集群分别执行以下命令，读取 Mesh v1 当前的 `meshId`：

```bash
kubectl -n istio-system get cm istio-1-22 -o jsonpath='{.data.mesh}'
```

从输出中找到 `meshId` 字段的取值；多集群网格的两个业务集群应当共享同一个 `meshId`，如果两侧取值不同，说明当前环境并非同一个 Mesh v1 多集群网格的两个成员，需要先厘清环境拓扑再继续判定。

接下来分别检查两个集群上是否存在东西向网关，并读取其 network 标签：

```bash
kubectl -n istio-system get svc istio-eastwestgateway -o jsonpath='{.metadata.labels.topology\.istio\.io/network}'
```

如果命令返回 `NotFound`，说明该集群没有部署东西向网关，记录为“无”；如果命令正常返回，记录输出的 network 取值。

最后读取每个集群 istiod 的 `CLUSTER_ID`，即该集群在多集群网格中使用的集群名：

```bash
kubectl -n istio-system get deploy istiod-1-22 -o jsonpath='{.spec.template.spec.containers[0].env[?(@.name=="CLUSTER_ID")].value}'
```

建议用下表记录两个集群的采集结果，供判定与后续迁移步骤使用：

| 采集项 | cluster1 | cluster2 |
|---|---|---|
| meshId | | |
| 东西向网关 network 标签 | | |
| clusterName（CLUSTER_ID） | | |

### 检查跨集群 Pod IP 直连连通性

在其中一个集群任选一个已注入 sidecar 的业务 Pod，获取另一个集群中任意业务 Pod 的 IP 地址：

```bash
# 在 cluster2 上下文执行，获取一个业务 Pod 的 IP
kubectl --kubeconfig <cluster2-kubeconfig> -n <mesh-namespace> get pod -o wide
```

回到 cluster1 上下文，从一个业务 Pod 内直接向上一步获取到的 cluster2 Pod IP 发起调用（替换为业务容器实际监听的端口）：

```bash
# 在 cluster1 上下文执行
kubectl --kubeconfig <cluster1-kubeconfig> -n <mesh-namespace> exec deploy/<workload-name> -- \
  curl -sf -o /dev/null -w '%{http_code}\n' --max-time 3 http://<cluster2-pod-ip>:<port>/
```

`<cluster1-kubeconfig>`/`<cluster2-kubeconfig>` 是本章新增的一对占位符，分别代表两个业务集群各自的 kubectl 访问配置；本章后续页面中，凡是需要明确区分两个业务集群的命令，都会使用这一对占位符，用法与全书既有的 `<global-kubeconfig>` 占位符一致。如果本地默认 kubectl 上下文已经是其中一个集群，对应的 `--kubeconfig` 参数可以省略。

命令返回预期的 HTTP 状态码，说明两个集群的 Pod 网络在底层已经直接互通；如果返回连接超时或不可达，说明底层网络并未打通，跨集群调用必须经过东西向网关转发。

### 对照判定表

将以上两步的结果对照下表，确定当前环境的网络拓扑：

| 判定特征 | 单网络拓扑 | 多网络拓扑 |
|---|---|---|
| 两集群 network 标签 | 均为“无”，或两侧取值相同 | 两侧均存在东西向网关，且取值不同 |
| 跨集群 Pod IP 直连 | 可达 | 不可达 |
| 东西向网关 | 不存在 | 两集群各自存在 |
| 对应迁移路径 | [迁移单网络多集群网格](./migrating-a-single-network-mesh.mdx) | [迁移多网络多集群网格](./migrating-a-multi-network-mesh.mdx) |

<Directive type="note" title="三项判定特征应当相互印证">
以上三项判定特征在实际环境中应当同时指向同一条路径；如果三项结果出现矛盾（例如存在东西向网关，但两侧 network 标签取值相同），请先联系 Alauda 支持确认环境的实际拓扑，再继续迁移。
</Directive>

</Steps>

判定完成后，按对照表进入对应的迁移路径页面。如果迁移完成后需要将单网络拓扑切换为多网络（例如为对齐产品认证形态），见[切换到多网络拓扑](./switching-to-multi-network.mdx)——该操作是迁移完成后的独立变更，不与迁移步骤混排。
