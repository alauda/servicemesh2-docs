# 迁移单网络多集群网格

本文对应[多集群网格迁移](./index.mdx)中判定为单网络拓扑的场景。本文在[单集群网格迁移](../migrating-a-single-cluster-mesh/index.mdx) P0 至 P6 六阶段流程的基础上，只描述单网络多集群场景相对多网络路径的差异点，不重复完整步骤；开始阅读前，请确认已经完成[迁移准备与前置检查](../preparing-for-migration.mdx)。

:::warning 本文步骤未经端到端实机演练
本文步骤基于已完整实测验证的多网络迁移基线推演适配。执行前，应先在与生产同形态的测试环境中完整演练一轮并确认结果符合预期。
:::

## 适用判定

本文适用于[多集群网格迁移](./index.mdx)“判定网络拓扑”一节判定为单网络拓扑的环境，判定特征归纳如下：

- 跨集群 Pod IP 可以直接互通（底层网络已经打平）；
- 两个集群均不存在东西向网关；
- 两个集群的 network 标签取值相同（含两侧均为“无”的情况）。

如果尚未完成判定，请先返回[多集群网格迁移](./index.mdx)执行判定步骤，确认当前环境确实属于单网络拓扑，再继续阅读本文；如果判定结果为多网络拓扑，应改为阅读[迁移多网络多集群网格](./migrating-a-multi-network-mesh.mdx)。

## 核心原则：迁移全程保持单网络配置不变

无论迁移进行到哪个阶段，以下红线贯穿全程，是本文相对多网络路径在原则上的唯一差异：

:::danger 严禁在迁移过程中变更网络拓扑配置
迁移全程，v2 的 network 相关配置必须逐字段对齐 v1 实测值。**严禁在迁移中途为 istio-system 命名空间添加 `topology.istio.io/network` 标签，或为 v2 单独设置 `values.global.network`。**否则两个控制面将对「端点属于哪个网络」产生认知分裂：v2 会把远端端点改写为尚未就位的东西向网关地址，造成跨集群路由黑洞。切换到多网络拓扑只能在迁移完成后作为独立变更进行，见 [切换到多网络拓扑](./switching-to-multi-network.mdx)。
:::

这条禁令背后的原理：network 的判定来源——`values.global.network` 配置项，以及 istio-system 命名空间与 Pod 上的 `topology.istio.io/network` 标签——在 v1、v2 两个控制面中语义相同，且 istio-system 命名空间本身是两个控制面共享的配置点。迁移中途改变这些配置，等于同时改变了 v1 控制面对拓扑的认知，而不只是 v2 一侧，两个控制面会在灰度过程中对同一批端点产生互相矛盾的判断。控制面换代（Mesh v1 到 Mesh v2）与网络拓扑变更（单网络到多网络）是两个相互独立的高风险变更，必须一次只做一个：本文的迁移步骤只处理前者，后者留到迁移完成后的独立变更窗口再执行，见下文“迁移完成后的拓扑选择”一节。

## 与多网络路径的差异

单网络多集群网格的迁移以[迁移多网络多集群网格](./migrating-a-multi-network-mesh.mdx)描述的多网络路径为基线，两者共享同一套单集群 P0–P6 流程与叠加逻辑，差异集中在以下五个阶段：

| 阶段 | 多网络路径 | 单网络变体 |
|---|---|---|
| P0 迁移准备 | 采集两集群 meshId/network/clusterName，以及东西向网关的 network 标签 | 同左；两集群 network 取值预期一致（含两侧均为“无”的情况）；不存在东西向网关，相应的采集与备份项相应减少；新增前置检查——跨集群 Pod IP 直连连通性，已经在[多集群网格迁移](./index.mdx)“判定网络拓扑”一节完成，无需重复执行 |
| P2 并存安装 v2 控制面 | `Istio` 资源的 `global` 追加 `meshID`/`network`/`multiCluster.clusterName` | 同左；`network` 逐字段照抄两集群在判定阶段记录的取值，分两种子情形：两侧均为「无」时，v2 的 `Istio` 资源不设置 `values.global.network`；两侧同名时，v2 在 `values.global.network` 写入与 v1 相同的同一取值（两集群一致）；共享 CA 与 remote secret 的复用逻辑不变——多网络路径已实测的这条复用逻辑，同样适用于单网络场景下的跨集群端点发现 |
| P2.5 部署东西向网关 | 每集群以 gateway injection 部署 v2 版东西向网关 | **整步跳过**：单网络拓扑下不存在东西向网关，没有对应的部署对象 |
| P3 迁移工作负载 | 跨集群三态调用矩阵（v1↔v1、v1↔v2、v2↔v2） | 矩阵验证方式相同，路径预期全程为 Pod IP 直连；新增单网络专属断言，见下文“单网络专属验证”一节 |
| P5 下线 Mesh v1 | 进入下线序列前，先统一摘除两集群东西向网关 Service 的归属标签 | 不适用：单网络拓扑没有东西向网关 Service；南北向网关 Service 的归属保护（摘除 `ownerReferences`）按[下线 Mesh v1](../migrating-a-single-cluster-mesh/decommissioning-mesh-v1.mdx)中的步骤照常执行 |

除以上五个阶段外，P1、P4、P6 三个阶段在两条路径下没有差异，均直接按[单集群网格迁移](../migrating-a-single-cluster-mesh/index.mdx)对应页面对两个集群分别执行；其余叠加内容（webhook 下线顺序、asm 自研 CR 的 finalizer 旁路处理、CA 交接、Kiali 与 VictoriaMetrics 对接等）与[迁移多网络多集群网格](./migrating-a-multi-network-mesh.mdx)完全同构，按该文对应章节执行即可，不需要额外调整。

## 单网络专属验证

除跨集群三态调用矩阵（与多网络路径验证方式相同）外，单网络拓扑还需要额外确认一点：跨集群流量的实际路径确实是 Pod 直连，而不是被某个组件转发。在已经切换到 v2 的命名空间中，任选一个业务 Pod，查看它对跨集群对端 Service 的 endpoint 解析结果：

```bash
istioctl proxy-config endpoints <pod-name> -n <mesh-namespace> | grep <peer-service>
```

**输出示例**

```
ENDPOINT                     STATUS      OUTLIER CHECK     CLUSTER
<cluster2-pod-ip>:<port>     HEALTHY     OK                outbound|<port>||<peer-service>.<mesh-namespace>.svc.cluster.local
```

`ENDPOINT` 列展示的地址应当是对端集群某个业务 Pod 的 IP（与[多集群网格迁移](./index.mdx)“判定网络拓扑”阶段验证直连时观察到的地址落在同一个 Pod 网段），而不是任何一侧东西向网关的地址——单网络拓扑下环境中本就不存在东西向网关。如果看到的地址明显不属于对端 Pod 网段，应当立即停止后续命名空间的切换，对照上文“核心原则”一节排查是否有配置在中途偏离了单网络约定。

## 迁移完成后的拓扑选择

完成 P0–P6 全部迁移步骤后，单网络拓扑的迁移即告完成，网格进入 Mesh v2 独占、且保持单网络配置的终态。是否需要在此基础上切换为多网络拓扑，取决于以下两点判断，本文不代您做出选择：

- 如果您需要将网格重新纳入产品 UI 管理，或者需要回到产品官方支持矩阵认证的拓扑形态，请在迁移完成后按照[切换到多网络拓扑](./switching-to-multi-network.mdx)执行独立的切换变更。
- 如果您可以接受手工维护原生 CR 的管理形态，并且看重单网络拓扑相对多网络的架构优势（跨集群调用少一跳东西向网关、少一组网关组件与对应的故障域），可以选择长期保持单网络终态；建议在做出决定前先联系 Alauda 支持确认单网络终态在您所用产品版本下的支持口径，再据此决定是否保留。

:::note 不要先在 Mesh v1 上切换为多网络，再开始迁移
即使已经确定终态需要多网络拓扑，也不要先在 Mesh v1 上完成单网络到多网络的切换，再开始本文描述的迁移流程：这相当于在即将整体下线的 v1 控制面上，对全网格的跨集群端点做一次高风险的重新计算，这份投入会在 v1 下线时被直接丢弃；而且 v1 的东西向网关是 asm 自研形态，进入迁移流程后仍然需要按[迁移网关](../migrating-a-single-cluster-mesh/migrating-gateways.mdx)与[下线 Mesh v1](../migrating-a-single-cluster-mesh/decommissioning-mesh-v1.mdx)的流程再迁移、再下线一遍，相当于把同一份网关迁移工作做了两次。正确的顺序是：先完成本文描述的单网络迁移，需要多网络时再按[切换到多网络拓扑](./switching-to-multi-network.mdx)在 Mesh v2 上使用上游标准做法完成切换，工具链与文档都更完整。
:::
