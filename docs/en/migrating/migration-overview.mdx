# 迁移概览

本文概述将 Alauda Service Mesh（Mesh v1）迁移到 Alauda Service Mesh v2（Mesh v2）的整体方案：为什么需要迁移、v1 与 v2 的架构和功能差异、迁移路径的选择、分阶段的迁移路线图，以及贯穿全书的占位符与告警约定。开始迁移前，请完整阅读本文档，再进入具体的操作页面。

:::info 版本适用性
本章节内容基于以下版本组合编写和验证：

| 组件 | 版本 |
|---|---|
| ACP | v4.3.x |
| Mesh v1 (ASM)（迁移源） | v4.3.5，istio 1.22.x |
| Mesh v2 (servicemesh-operator2)（迁移目标） | v2.1.x，istio 1.28.x |

如果您的环境版本与上表不同，请先联系 Alauda 支持确认适用性，再参考本章节执行迁移。
:::

## 为什么要迁移到 Mesh v2

Mesh v1（Alauda Service Mesh）基于社区已经废弃的 istio-operator 项目构建。ACP 计划在 v4.4 版本中下线 Mesh v1；后续的服务网格能力由 Mesh v2（servicemesh-operator2）提供，其控制面基于社区持续维护的 Sail Operator 项目构建。使用 Mesh v1 的客户需要在 ACP v4.4 发布之前，将存量网格迁移到 Mesh v2，以便继续获得功能更新与技术支持。

## 架构对比

下表对比 Mesh v1 与 Mesh v2 在关键维度上的实现差异：

| 维度 | Mesh v1 (ASM) | Mesh v2 (servicemesh-operator2) |
|---|---|---|
| 控制面 | 基于社区已废弃的 istio-operator；`IstioOperator` CR 驱动部署 istiod（revision `1-22`） | 基于 Sail Operator；`Istio`/`IstioRevision` CR 驱动部署，`updateStrategy` 为 `RevisionBased` |
| 注入机制 | `istio-init` 容器执行 iptables 流量劫持 | `IstioCNI` + native sidecar（无需 `istio-init`） |
| 网关形态 | 自研 `GatewayDeploy`/`IngressGateway` CR，每个网关对应一个 `IstioOperator` CR | Gateway injection Deployment（自行管理）或 Gateway API |
| 可观测栈 | 自研 OTel v1 采集链路 + Jaeger v1（Elasticsearch）+ 自研调用链/拓扑 UI | OTel v2 + Jaeger v2（`jaeger-system`）+ Kiali |
| 多集群 | `ServiceMeshGroup` CR + 平台自研控制器编排 | 原生 multi-primary/primary-remote 拓扑，按需手工配置 |
| 管理入口 | 平台 UI + 自研 CR（`ServiceMesh`/`MicroService` 等） | YAML/kubectl 直接操作原生 CR（`Istio`/`IstioRevisionTag` 等） |

## 功能对照与替代方案

Mesh v1 的部分能力由平台自研 CR 实现，这些 CR 在 Mesh v2 中不再提供；下表汇总每项能力在 Mesh v2 中的对应方式与迁移动作：

| v1 能力 | v1 实现方式 | v2 对应方式 | 迁移动作 |
|---|---|---|---|
| 网格创建与管理 | `ServiceMesh` CR + 平台 UI | `Istio`/`IstioCNI` CR（YAML/kubectl 管理） | 见下文迁移路线图 P2 阶段 |
| 微服务纳管 | `MicroService` CR + UI | 无对应概念，使用原生命名空间/Pod 标签 | 工作负载切换后，v1 自研 CR 随 P5 一并下线 |
| Sidecar 注入 | 命名空间 `istio.io/rev=1-22` 标签 + `cpaas.io/serviceMesh` webhook | 命名空间 `istio.io/rev=<v2-revision>` 标签（终态可选收编为 `istio-injection=enabled`） | P3 阶段 |
| 入口/出口网关 | 自研 `GatewayDeploy`/`IngressGateway` CR，每个网关对应一个 `IstioOperator` CR | Gateway injection Deployment 或 Gateway API | P4 阶段 |
| `Gateway`/`VirtualService`/`DestinationRule`/`ServiceEntry`/`Sidecar`/`PeerAuthentication` 等 istio 原生资源 | 原生资源 | 原样复用 | 需在 P0 阶段完成 istio 1.28 schema 兼容性检查 |
| 流量策略（熔断、连接池、负载均衡） | `ConnectionPool`/`OutlierDetection`/`LoadBalancer` 自研 CR，生成原生 `DestinationRule` | 直接维护原生 `DestinationRule` | 确认自动生成的原生资源已保留后，删除自研 CR |
| 灰度发布 | `CanaryDelivery`（内置 fork 版 Flagger） | 无内置能力；可集成 ACP <ExternalSiteLink name="acp" href="/developer/building_application/operation_maintaining/application_rollouts/canary_deployment.html" children="Argo Rollouts" /> 实现渐进式发布 | 存量 Canary 发布必须在迁移前终止 |
| 全局限流 | `GlobalRateLimiter` + 自研限流服务 + Redis | 无内置能力；采用 [istio 官方 rate limit 方案](https://istio.io/latest/docs/tasks/policy-enforcement/rate-limit/)（EnvoyFilter + RLS 服务） | 迁移前评估存量限流规则的等价替代方案 |
| 本地限流 | istio 官方 EnvoyFilter 方案 | 相同 | 原样保留，注意 istio 1.28 兼容性 |
| 安全策略（黑白名单、JWT） | `Whitelist`/`JWTPolicy` 自研 CR | 原生 `AuthorizationPolicy`/`RequestAuthentication` | 转换为原生资源后删除自研 CR |
| 实例隔离 | `IsolatePod`（readinessGate） | 无直接对应能力 | 需按业务场景手工摘除 endpoints/标签，实现等价效果 |
| API 管理/API 流量监控 | `APIAttribute` + WasmPlugin | 无直接对应能力；可参考 istio 官方 [Classifying Metrics](https://istio.io/latest/docs/tasks/observability/metrics/classify-metrics/) 方案 | 按业务需要评估重建 |
| 调用链/拓扑/监控 UI | 自研调用链 UI 与拓扑聚合服务 | Kiali（及 Jaeger UI） | P1、P6 阶段 |
| 调用链采集管道 | OTel v1 + Jaeger v1（Elasticsearch） | OTel v2 + Jaeger v2（`jaeger-system`） | 见[迁移可观测栈](./migrating-observability.mdx) |
| 多集群编排 | `ServiceMeshGroup` CR + 平台自研控制器 | 原生 multi-primary/primary-remote 拓扑 | 见[多集群网格迁移](./migrating-a-multi-cluster-mesh/index.mdx) |
| `ASM_MS_NAME` 环境变量 / `asm.cpaas.io/msname` 标签 | 自研环境变量/标签，用于标识微服务身份 | 不再提供，使用标准的 `app`/canonical service 标签体系 | 业务侧感知项：如应用或配置依赖该环境变量或标签，需在迁移前完成适配 |

## 迁移策略选择

Mesh v2 迁移提供两种路径：

- **并存灰度迁移（推荐）**：v2 控制面与 v1 控制面在同一集群内并存运行，工作负载与网关按命名空间/服务粒度灰度切换，业务流量在迁移过程中不中断。该路径已在单集群与多集群场景下完成全流程验证，是本章节的主线内容。
- **停机卸载重装迁移**：先卸载 Mesh v1，再全新安装 Mesh v2，全程网格流量与遥测均会中断，仅适用于可接受停机的测试或非生产环境。适用条件、限制与操作步骤见[停机卸载重装迁移](./migrating-with-downtime.mdx)。

如无特殊原因（例如需要快速重建的测试环境），应优先选择并存灰度迁移路径，本文档后续章节均围绕该路径展开。

## 迁移路线图

并存灰度迁移划分为 7 个阶段，编号 P0–P6；本章节后续所有页面均使用这套阶段术语：

| 阶段 | 说明 | 回滚点 | 相关文档 |
|---|---|---|---|
| P0 迁移准备 | 盘点网格资源、创建备份、完成兼容性审计、约定变更冻结、部署业务探测器 | 可回滚 | [迁移准备与前置检查](./preparing-for-migration.mdx) |
| P1 准备可观测性 | 安装 Kiali 并接入现有监控数据源，创建 v2 的指标采集资源 | 可回滚 | [安装 v2 控制面](./migrating-a-single-cluster-mesh/installing-the-v2-control-plane.mdx) |
| P2 并存安装 v2 控制面 | 安装 servicemesh-operator2，创建 IstioCNI 与 Istio CR，与 Mesh v1 控制面并存运行、对其零扰动 | 可回滚（删除 v2 `Istio`/`IstioCNI` CR） | [安装 v2 控制面](./migrating-a-single-cluster-mesh/installing-the-v2-control-plane.mdx) |
| P3 迁移工作负载 | 按命名空间将 sidecar 注入切换到 v2 修订，验证后再处理下一个命名空间 | 可回滚（标签改回 + 滚动重启） | [迁移工作负载](./migrating-a-single-cluster-mesh/migrating-workloads.mdx) |
| P4 迁移网关 | 在网关命名空间部署 v2 金丝雀网关，验证后将流量从 v1 旧网关切换过去 | 可回滚（v1 旧网关在 P5 之前始终保留） | [迁移网关](./migrating-a-single-cluster-mesh/migrating-gateways.mdx) |
| P5 下线 Mesh v1 | 按手工序列下线 Mesh v1 的全部组件 | **不可回退点** | [下线 Mesh v1](./migrating-a-single-cluster-mesh/decommissioning-mesh-v1.mdx) |
| P6 完成迁移 | 收编 default 修订、完成 CA 生命周期交接、切换可观测性到最终形态 | 不适用（Mesh v1 已下线） | [完成迁移](./migrating-a-single-cluster-mesh/finalizing-the-migration.mdx) |

P5 是本方案中唯一的不可回退点：进入该阶段前，必须确认对应的 Go/No-Go 检查单全部通过。P5 之后如需恢复 Mesh v1，只能按备份执行灾难恢复级重装；P0–P4 各阶段的具体回滚动作与 P5 之后的恢复清单，见[回滚指引](./rollback.mdx)。

以上路线图以单集群场景为基线；多集群网格在 P0、P2–P6 阶段有叠加步骤（网络拓扑判定、跨集群证书与 remote secret 复用、东西向网关切换、每集群独立中间 CA 与可观测收尾等），详见[多集群网格迁移](./migrating-a-multi-cluster-mesh/index.mdx)。

## 并存为什么是安全的

在 P2–P4 阶段，v1 与 v2 两个控制面会在同一个集群内并存一段时间。以下四点说明了为什么这种并存在满足既定操作顺序的前提下是安全的：

- **同根 CA**：v1 的 `cacerts`（位于 istio-system 命名空间）本身就是标准的 istio 插件 CA 格式。v2 istiod 部署到同一命名空间后会直接加载这份 `cacerts` 作为签发 CA，因此新旧数据面签发的工作负载证书同根，mTLS 在迁移全程保持互通。
- **revision 隔离**：v1 本身就是以 istio 原生 revision 机制部署的（revision 名为 `1-22`），与 v2 的 revision（例如 `default-v1-28-6`）属于同一套多修订并存模型。命名空间的 `istio.io/rev` 标签是精确匹配，改标签即完成注入归属切换，两个控制面不会同时向同一个命名空间注入 sidecar。
- **webhook 错峰**：v1 占用了 default 位置的 webhook 配置。只要并存期间不为 v2 创建 default 的 `IstioRevisionTag`，v2 的 webhook 就都带 revision 后缀，与 v1 的 webhook 互不重叠。
- **CNI 互斥**：v1 数据面使用 `istio-init` 容器完成流量劫持，v2 使用 `IstioCNI`。上游 istio-cni 对包含 `istio-init` 容器的 Pod 会自动跳过处理，因此两种注入方式可以在同一节点混跑而不冲突。

## 重要限制

以上安全性论证的前提是并存期间遵守既定的操作顺序。以下红线将贯穿整个迁移过程，会在[安装 v2 控制面](./migrating-a-single-cluster-mesh/installing-the-v2-control-plane.mdx)页末与[下线 Mesh v1](./migrating-a-single-cluster-mesh/decommissioning-mesh-v1.mdx)页首再次强调：

:::danger 严禁通过平台 UI 删除 Mesh v1 网格
**v2 `Istio` CR 创建后，严禁通过平台 UI 删除 Mesh v1 网格，或删除 `ServiceMesh` CR 后等待控制器处理。** Mesh v1 的删除流程会删除整个 istio-system 命名空间，v2 控制面与网格配置将随之一并被摧毁。Mesh v1 的下线必须按照 [下线 Mesh v1](./migrating-a-single-cluster-mesh/decommissioning-mesh-v1.mdx) 的手工序列执行。
:::

## 约定

本章节所有页面遵循下列约定。

**占位符**：正文中出现的尖括号占位符代表需要替换为您实际环境取值的参数，不是可以直接复制执行的字面量。全书通用的占位符如下，个别页面专用的占位符会在使用处单独说明：

| 占位符 | 含义 | 发现命令（正文按需给出） |
|---|---|---|
| `<cluster-name>` | 业务集群名 | — |
| `<v2-revision>` | v2 活动修订名（示例 `default-v1-28-6`） | `kubectl get istiorevisions.sailoperator.io` |
| `<mesh-namespace>` | 参与网格的业务命名空间 | `kubectl get ns -l istio.io/rev=1-22` |
| `<gateway-namespace>` / `<gateway-name>` | 网关所在 ns / 网关名 | `kubectl get gatewaydeploys.asm.alauda.io -A` |
| `<control-plane-iop>` | v1 控制面 IstioOperator CR 名 | `kubectl -n istio-system get istiooperators.install.istio.io` |
| `<global-kubeconfig>` | global 集群访问配置 | — |
| `<network-1>` / `<network-2>` | 多网络模式下两集群 network 名 | 见拓扑判定采集命令 |

**发现命令**：凡是环境特定的取值（例如具体的 meshID、网关 Service selector），正文都会在使用前给出一条读取该值的命令（通常是 `kubectl get`/`kubectl -o jsonpath` 查询），而不是要求您凭经验猜测。执行迁移步骤前，请先运行对应的发现命令，确认取值后再替换到后续命令中。
