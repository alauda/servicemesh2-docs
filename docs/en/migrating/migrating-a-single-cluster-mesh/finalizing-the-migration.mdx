# 完成迁移（收编与 CA 交接）

本文对应[迁移概览](../migration-overview.mdx)路线图中的 P6 完成迁移阶段：Mesh v1 下线完成后，把并存灰度期间遗留的少量 v1 时期资源正式收编为 Mesh v2 的标准形态，完成证书颁发机构（CA）的生命周期交接，并确认可观测性已经切换到最终形态，使集群进入完全自持、不再依赖任何 Mesh v1 组件的终态。开始操作前，请确认已经完成[下线 Mesh v1](./decommissioning-mesh-v1.mdx)中的全部步骤与验证点，业务集群中已经不存在任何 Mesh v1 组件。下文的 default 修订收编、CA 生命周期交接、可观测终态三部分之间没有强制的先后顺序，可以按需并行或交替推进，全部完成后用终态验收清单确认迁移正式结束。

## default 收编

Mesh v1 下线阶段已经删除了占用 default 位置的 webhook 配置（见[下线 Mesh v1](./decommissioning-mesh-v1.mdx)「清理残留标签、webhook 与 CRD」一节），default 位置现在处于空闲状态。本节把它正式交还给 Mesh v2：为 Mesh v2 创建 default 修订的 `IstioRevisionTag`，使集群可以使用最简单的 `istio-injection=enabled` 标签完成注入，不必再记忆具体的 `<v2-revision>` 取值。

<Steps>

### 补齐 Helm 元数据

Mesh v1 的 istio-operator 在最初部署时直接创建了 `istio-reader-service-account` 等一批基础资源，这些资源不带任何 Helm 归属标记。Mesh v2 的 default 修订由 sail-operator 的 `base` chart 管理；创建 default 修订的 `IstioRevisionTag` 会触发 sail-operator 协调这批基础资源，如果发现同名资源已经存在但缺少 Helm 归属信息，会判定为归属冲突而拒绝接管，导致 `IstioRevisionTag` 停留在 `ReconcileError` 状态。

为避免这个冲突，先给 `istio-reader-service-account` 补齐 Helm 归属的标签与注解：

```bash
kubectl -n istio-system label sa istio-reader-service-account app.kubernetes.io/managed-by=Helm
kubectl -n istio-system annotate sa istio-reader-service-account meta.helm.sh/release-name=default-base meta.helm.sh/release-namespace=sail-operator
```

如果完成下一步创建 `IstioRevisionTag` 之后仍然报 `ReconcileError`，说明当前环境还存在其他 v1 时期遗留的、同类未受 Helm 管理的基础资源：查看错误信息中提到的具体资源名称，用同样的方法为其补齐 Helm 归属的标签与注解。

### 创建 default IstioRevisionTag

```yaml
apiVersion: sailoperator.io/v1
kind: IstioRevisionTag
metadata:
  name: default
spec:
  targetRef:
    kind: Istio
    name: default
```

将以上内容保存为 `istioRevisionTag.yaml` 后执行：

```bash
kubectl apply -f istioRevisionTag.yaml
```

验证：

```bash
kubectl get istiorevisiontags.sailoperator.io
```

**输出示例**

```
NAME      STATUS    IN USE   REVISION          AGE
default   Healthy   True     default-v1-28-6   15s
```

`REVISION` 列应指向当前的 `<v2-revision>`；`STATUS`/`IN USE` 列的取值含义见下一步说明。

### （可选）回归 istio-injection=enabled 注入模型

是否执行本步骤属于可选项：继续使用 `istio.io/rev=<v2-revision>` 标签作为长期的注入方式同样受支持，两种终态在功能上没有差异，可以按团队的标签管理习惯选择。如果选择回归到最简单的 `istio-injection=enabled` 模型，对每个已完成迁移的业务命名空间执行：

```bash
kubectl label ns <mesh-namespace> istio-injection=enabled istio.io/rev-
kubectl -n <mesh-namespace> rollout restart deploy
```

命令同时补上 `istio-injection=enabled` 标签并摘除 `istio.io/rev` 标签，避免两个标签同时存在造成注入归属歧义；随后的滚动重启使已运行的 Pod 切换到通过 default tag 注入的路径。完整原理、逐步验证与操作细节见 [Enabling sidecar injection with IstioRevisionTag resource](../../installing/sidecar-injection/sidecar-injection-with-istiorevisiontag.mdx)。

<Directive type="note" title="未执行本步骤时，IstioRevisionTag 的 STATUS 会显示 NotReferencedByAnything">
如果保留 `istio.io/rev` 标签、不执行本步骤，`kubectl get istiorevisiontags.sailoperator.io` 中 `default` 这一行的 `STATUS` 会持续显示 `NotReferencedByAnything` 而不是 `Healthy`——这同样是正常状态，代表这个 tag 存在但当前没有命名空间通过 `istio-injection=enabled` 引用它，不代表配置有误。只有当至少一个命名空间执行了本步骤之后，`STATUS` 才会变为 `Healthy`。
</Directive>

</Steps>

## CA 生命周期交接

Mesh v1 的证书体系是分层结构：根证书有效期极长、长期稳定不变；每个业务集群实际使用的中间证书有效期只有 60 天，依赖 `global-asm-controller` 在到期前自动续期并同步到业务集群的 `cacerts`。[下线 Mesh v1](./decommissioning-mesh-v1.mdx)阶段已经彻底移除了 `global-asm-controller`，这条自动续期链路随之消失——业务集群当前使用的中间证书不会再自动续期，放任不管会在到期后导致数据面证书签发失败。根证书本身不受影响，不需要更换；本节的目标是用同一份根证书签发一张新的长期中间证书，把证书生命周期的管理权正式交接给日常运维流程。

<Steps>

### 导出根证书

根证书 `asm-istio-rootca`（`kubernetes.io/tls` 类型 Secret）保存在 global 集群的 cpaas-system 命名空间。[迁移准备与前置检查](../preparing-for-migration.mdx)阶段的备份中已经包含它的完整 YAML（`$BACKUP_DIR/asm-istio-rootca.yaml`）；这里从中提取签发新证书所需的证书与私钥文件：

```bash
# 在 global 集群上下文执行
kubectl --kubeconfig <global-kubeconfig> -n cpaas-system get secret asm-istio-rootca -o jsonpath='{.data.tls\.crt}' | base64 -d > root-cert.pem
kubectl --kubeconfig <global-kubeconfig> -n cpaas-system get secret asm-istio-rootca -o jsonpath='{.data.tls\.key}' | base64 -d > root-key.pem
```

<Directive type="warning" title="root-key.pem 离线保管，不落业务集群磁盘">
`root-key.pem` 是整条信任链的根私钥，一旦泄露，持有者可以签发被网格无条件信任的任意证书。导出后只在受控的本地工作站或专用签发环境中短暂使用，仅用于下一步签发新的中间证书；签发完成后立即从本地磁盘删除该文件，不要以任何形式落盘到业务集群、上传到代码仓库或制品库。`root-cert.pem`（证书公钥部分）不敏感，可以正常保留用于后续构建证书链。
</Directive>

### 签发长期中间证书

按照[多集群配置概览](../../installing/multi-cluster/configuration-overview.mdx)「[Creating certificates for a multi-cluster mesh](../../installing/multi-cluster/configuration-overview.mdx#creating-certificates-for-a-multi-cluster-mesh)」一节中签发中间证书的完整 openssl 流程执行，与该流程的差异只有两点：

- 不需要执行该节的「Create the root CA certificate」部分：签发中间证书所需的 `root-key.pem`、`root-cert.pem` 直接使用上一步导出的文件，从「Create the intermediate CA certificate for the **East**/**West** cluster」部分对应本集群的取值开始执行即可；
- 该流程示例中的中间证书有效期为 `-days 3650`（10 年）；建议按客户证书策略调整为 1–2 年（例如 `-days 730`），并建立到期前的轮换 SOP——到期前手工重复本节与下一步的流程，重新签发并替换 `cacerts` 即可完成一次轮换，不需要为此额外部署自动化续期组件。

流程执行完成后，应当得到 `ca-cert.pem`、`ca-key.pem`、`root-cert.pem`、`cert-chain.pem` 四个文件，用于下一步重建 `cacerts`。

### 重建 cacerts

istio-system 命名空间下已经存在名为 `cacerts` 的 Secret——这正是并存灰度迁移期间 Mesh v1、Mesh v2 两个控制面共用的那一份 CA（见[安装 v2 控制面](./installing-the-v2-control-plane.mdx)）。用新签发的四个文件原地覆盖它：

```bash
kubectl -n istio-system create secret generic cacerts \
  --from-file=ca-cert.pem=ca-cert.pem \
  --from-file=ca-key.pem=ca-key.pem \
  --from-file=root-cert.pem=root-cert.pem \
  --from-file=cert-chain.pem=cert-chain.pem \
  --dry-run=client -o yaml | kubectl apply -f -
```

istiod 监听 `cacerts` 的变化并自动热加载新的签发证书，不需要重启 istiod 或任何数据面 Pod。

### 验证热加载

确认 istiod 已经加载新的 `cacerts`：

```bash
kubectl -n istio-system logs deploy/istiod-<v2-revision> | grep "Update Istiod cacerts"
```

期望日志中出现 `Update Istiod cacerts`（以及紧随其后的 `update root cert and generate new dns certs`），说明 istiod 已经完成热加载。

确认存量 Pod 没有因为这次更新发生非预期重启：

```bash
kubectl get pods -n <mesh-namespace>
```

对比更新前的记录，`RESTARTS` 计数与 `AGE` 均不应发生变化。

滚动重启一个工作负载，验证新证书链可以正常签发：

```bash
kubectl -n <mesh-namespace> rollout restart deploy/<workload-name>
kubectl -n <mesh-namespace> rollout status deploy/<workload-name>
```

从 `kubectl -n <mesh-namespace> get pods` 的输出中找到新建 Pod 的名称，核实其证书的签发者（issuer）为新的中间证书：

```bash
istioctl proxy-config secret <pod-name> -n <mesh-namespace> -o json \
  | jq -r '.dynamicActiveSecrets[] | select(.name=="default") | .secret.tlsCertificate.certificateChain.inlineBytes' \
  | base64 -d | openssl x509 -noout -issuer
```

期望输出中的 issuer 字段与本次签发时 `intermediate.conf` 里 `[ req_dn ]` 的取值一致，不再是旧的中间证书信息。

<Directive type="note" title="存量工作负载证书不会立即轮换，属预期行为">
`cacerts` 更新后，istiod 只对新建立的证书请求签发新证书；存量 Pod 已经持有的工作负载证书在剩余有效期内（默认 TTL 为 24 小时）不会被主动吊销或替换。因此重建 `cacerts` 之后的短时间内（例如 90 秒内）检查存量 Pod，看到的证书 issuer 大多仍然是旧的中间证书，这是正常现象，不代表热加载失败，这些证书会在各自到期、重新申请时自然切换到新的证书链。新旧中间证书共享同一个根证书，在这段混存期内，新旧证书签发的工作负载之间的 mTLS 互通不受影响。如果需要立即确认新证书链是否可以正常签发，滚动重启单个 Deployment（如上文命令）即可。
</Directive>

</Steps>

## 可观测终态

Mesh v1 下线阶段已经随带移除了 v1 时期的 OTel Collector、Jaeger v1 与自研调用链/拓扑 UI。Mesh v2 可观测栈的终态收尾——安装 OTel v2 组件与 Jaeger v2、将 Istio CR 的 `extensionProviders` 切换到新的 Collector、迁移 OTel Java 自动注入、把 Kiali 的调用链数据源切换到 Jaeger v2——见[迁移可观测栈](../migrating-observability.mdx)。

时序上，可观测栈的终态切换可以在完成[下线 Mesh v1](./decommissioning-mesh-v1.mdx)之后随时进行，与本文的 default 收编、CA 生命周期交接两节没有强制的先后依赖，三者都是 P6 阶段内彼此独立的收尾工作。进入下文的终态验收之前，需要确认三部分均已完成。

## 终态验收

完成以上全部工作后，确认以下检查点。

`Istio`、`IstioCNI`、`IstioRevisionTag` 三类资源均应处于健康状态：

```bash
kubectl get istio,istiocni,istiorevisiontag
```

**输出示例**

```
NAME                            NAMESPACE      PROFILE   REVISIONS   READY   IN USE   ACTIVE REVISION   STATUS    VERSION   AGE
istio.sailoperator.io/default   istio-system             1           1       1        default-v1-28-6   Healthy   v1.28.6   3h

NAME                               NAMESPACE   PROFILE   READY   STATUS    VERSION   AGE
istiocni.sailoperator.io/default   istio-cni             True    Healthy   v1.28.6   3h

NAME                                       STATUS    IN USE   REVISION          AGE
istiorevisiontag.sailoperator.io/default   Healthy   True     default-v1-28-6   1h
```

如果「default 收编」一节中的可选步骤没有执行、选择长期保留 `istio.io/rev` 标签，`istiorevisiontag` 的 `STATUS` 会显示 `NotReferencedByAnything` 而不是 `Healthy`，这同样是合规的终态，不需要为了改变这一状态而强行执行可选步骤。

打开 Kiali 控制台，在命名空间选择器中选择全部业务命名空间，进入 **Graph** 页面，确认拓扑中显示的服务与调用关系与实际业务架构一致，不再出现任何 Mesh v1 时期的组件。完成[迁移可观测栈](../migrating-observability.mdx)后，在 Kiali 中触发一次调用，确认可以在 Jaeger v2 中查询到对应的调用链。

重新执行一次残留扫描，确认本文的操作没有引入任何新的残留或非预期副作用：

```bash
# 以下三条命令均应返回 0 或无输出
kubectl get crd | grep -c asm.alauda.io
kubectl -n istio-system get deploy | grep -Ei "istiod-1-22|istio-operator|asm-operator|flagger-operator|jaeger-operator|opentelemetry-operator-controller-manager|istio-ingressgateway"
kubectl --kubeconfig <global-kubeconfig> -n cpaas-system get deploy | grep -Ei "asm|hermes|mephisto"
```

全部检查点通过后，进行以下收尾动作：

- 停止[迁移准备与前置检查](../preparing-for-migration.mdx)阶段部署的业务探测器进程，将全程探测日志与本文各步骤的验证输出一并归档，作为迁移完整性的凭证；
- 将 `$BACKUP_DIR` 中的备份物料转移到长期归档存储，按内部数据保留策略保存；确认迁移全流程验证通过后，不需要再在业务集群本地保留大体量的备份文件；
- 通知干系人变更冻结解除，Mesh v1 到 Mesh v2 的迁移正式完成，网格恢复正常的变更与发布流程。

至此，Mesh v1 到 Mesh v2 的单集群迁移全部完成。

:::tip 多集群场景
default 修订收编与 CA 生命周期交接需要对每个业务集群单独执行一次。根证书全局只有一份，从 global 集群导出一次即可复用；多集群场景建议为每个业务集群签发独立的中间证书（可以用 `intermediate.conf` 中的 `L` 字段区分集群，例如 `L=<cluster-name>`），全部使用同一份根证书签发——只要根证书一致，跨集群 mTLS 不受影响。可观测终态与终态验收同样需要对每个业务集群分别确认。见[多集群网格迁移](../migrating-a-multi-cluster-mesh/index.mdx)。
:::
